Data Sources / Persistence

    SQL Databases

    • SQL databases, short for Structured Query Language databases, are a type of relational database management system (RDBMS) that use SQL as their query language.

    Key aspects of SQL databases include:

    Tables:

    • Tables are the core structures in a SQL database and consist of rows and columns where data is stored.
    • Each table represents a specific type of data, such as customers, orders, or products.

    Schema:

    • A schema defines the structure of the database, including tables, columns, data types, and relationships between tables.

    Primary Key:

    • A unique identifier for each record in a table. It ensures that each row can be uniquely identified.

    Composite Key:

    • A composite key is a primary key or unique key that consists of multiple columns. It ensures that the combination of values in the specified columns is unique across all rows.

    Foreign Key:

    • A column that creates a link between two tables. It is used to maintain referential integrity between the tables.

    SQL Queries:

    • SELECT: Used to retrieve data from one or more tables.
    • INSERT: Used to add new records to a table.
    • UPDATE: Used to modify existing records.
    • DELETE: Used to remove records from a table.
    • JOIN: Combines rows from two or more tables based on a related column between them.

    • Open-source database widely used in web applications.
    • Known for its reliability and ease of use.
    • Commonly used with PHP in the LAMP stack (Linux, Apache, MySQL, PHP).

    PostgreSQL:

    • Advanced open-source database known for its robustness, extensibility, and standards compliance.
    • Supports complex queries, foreign keys, triggers, and user-defined types.

    SQLite:

    • A lightweight, embedded SQL database engine.
    • Ideal for mobile apps, small to medium websites, and applications where simplicity and minimal setup are important.

    SQL

    • SQL (Structured Query Language) is a standard language used to interact with relational databases. It allows you to create, read, update, and delete data (CRUD operations), as well as manage database structures.

    Key Features of SQL

    • Data Querying: Retrieve data using SELECT statements.
    • Data Manipulation: Insert, update, and delete data (INSERT, UPDATE, DELETE).
    • Schema Definition: Define database structures (CREATE TABLE, ALTER TABLE).
    • Data Control: Manage access permissions (GRANT, REVOKE).
    • Transaction Management: Ensure data integrity (COMMIT, ROLLBACK).

    SQL syntax examples:

    Database Operations:

    • Create a database:
      CREATE DATABASE my_database;
      CREATE DATABASE IF NOT EXISTS my_database;
    • Use a database:
      USE my_database;
    • Drop a database:
      DROP DATABASE my_database;
      DROP DATABASE IF EXISTS my_database;

    Table Operations:

    • Show tables:
      CREATE TABLE users (
      id INT AUTO_INCREMENT PRIMARY KEY,
      name VARCHAR(100) NOT NULL,
      email VARCHAR(255) UNIQUE NOT NULL,
      created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP);
    • Show tables:
      SHOW TABLES;
    • Describe a table structure:
      DESC users;
    • Drop a table:
      DROP TABLE users;

    Data Manipulation (DML):

    • Insert data:
      INSERT INTO users (name, email) VALUES ('John Doe', 'john@example.com');
    • Select data:
      SELECT * FROM users;
    • Select with conditions:
      SELECT * FROM users WHERE email = 'john@example.com';
    • Update data:
      UPDATE users SET name = 'Jane Doe' WHERE id = 1;
    • Delete data:
      DELETE FROM users WHERE id = 1;

    Constraints:

    • PRIMARY KEY - Ensures a column (or combination of columns) uniquely identifies each row. Automatically applies UNIQUE and NOT NULL constraints. Each table can have only one or more primary key.
      CREATE TABLE Products (
      id INT PRIMARY KEY,
      name VARCHAR(100) NOT NULL);

      CREATE TABLE orders (
      order_id INT,
      product_id INT,
      quantity INT,
      PRIMARY KEY (order_id, product_id));
    • FOREIGN KEY - Establishes a relationship between two tables. Ensures referential integrity by restricting invalid data entries.
      CREATE TABLE Orders (
      order_id INT PRIMARY KEY,
      product_id INT,
      FOREIGN KEY (product_id) REFERENCES Products(id));
      Supports ON DELETE CASCADE or ON DELETE SET NULL.
    • UNIQUE - Ensures all values in a column are distinct. Unlike PRIMARY KEY, multiple UNIQUE constraints can exist per table.
      CREATE TABLE Users (
      user_id INT PRIMARY KEY,
      email VARCHAR(255) UNIQUE );
    • NOT NULL - Prevents a column from storing NULL values.
      CREATE TABLE Products (
      id INT PRIMARY KEY,
      name VARCHAR(100) NOT NULL);
    • CHECK - Ensures a column satisfies a condition.
      CREATE TABLE Products (
      id INT PRIMARY KEY,
      price DECIMAL(10,2) CHECK (price > 0) );
    • DEFAULT - Assigns a default value if no value is provided.
      CREATE TABLE Orders (
      id INT PRIMARY KEY,
      status VARCHAR(20) DEFAULT 'Pending' );
    • AUTO_INCREMENT - Automatically generates a unique number for each row.
      CREATE TABLE Users (
      id INT AUTO_INCREMENT PRIMARY KEY,
      name VARCHAR(100) NOT NULL );
    • ENUM - Restricts values to a predefined list.
      CREATE TABLE Employees (
      id INT PRIMARY KEY,
      role ENUM('Admin', 'Manager', 'Staff') NOT NULL );

    Joins:

    Indexes:

    • Create an index:
      CREATE INDEX idx_email ON users(email);
    • Drop an index:
      DROP INDEX idx_email ON users;

    Transactions:

    • Commit a transaction:
      START TRANSACTION;
      UPDATE users SET name = 'Alice' WHERE id = 2;
      COMMIT;
    • Rollback a transaction:
      START TRANSACTION;
      UPDATE users SET name = 'Alice' WHERE id = 2;
      ROLLBACK;

    Database Design

    • Database design is the process of structuring and organizing data to ensure efficiency, consistency, and scalability. It involves defining tables, relationships, constraints, and indexes to optimize data storage and retrieval. Key steps include:

    Database Design Steps:

    • Requirement Analysis: Understanding data needs.
    • Conceptual Design: Creating an entity-relationship diagram (ERD).
    • Logical Design: Defining tables, columns, data types, and relationships.
    • Normalization: Reducing redundancy by structuring data into related tables.
    • Physical Design: Optimizing performance with indexing, partitioning, and storage strategies.
    • Implementation and Testing: Creating the database, inserting data, and validating queries.

    What is an Entity?

    An entity in a database represents a real-world object or concept that is stored as a table. Each entity typically has attributes (columns) and instances (rows).

    For example, in an e-commerce application:

    • A Product entity could have attributes like id, name, and price.
    • A Customer entity could have attributes like id, name, and email.

    What is an Entity Relationship?

    Refers to a conceptual model that represents data and its relationships within a system. It is commonly used in database design to visually map how entities (objects) relate to each other.

    Types of Entity Relationship:

    • One-to-One (e.g., a User has one Profile)
    • One-to-Many (e.g., a Customer has many Orders)
    • Many-to-Many (e.g., Students enrolled in multiple Courses)

    What is Normalization?

    Normalization in databases is the process of organizing data to reduce redundancy and improve data integrity. It involves structuring tables to ensure that each piece of data is stored in only one place, which minimizes anomalies during insert, update, and delete operations.

    Pros of Normalization:

    • Eliminates Data Redundancy - Reduces storage space by avoiding duplicate data across tables.
    • Ensures Data Integrity - Changes made to data in one place reflect everywhere, reducing inconsistencies.
    • Improves Data Consistency - Since data is stored in only one place, there’s no risk of conflicting information.
    • Easier to Maintain - Updating, deleting, and inserting data is simpler and less error-prone.
    • Better Scalability - Structured data makes it easier to scale databases as the application grows.
    • Reduces Anomalies - Prevents insertion, update, and deletion anomalies, making the database more reliable.
    • Improved Query Optimization (in some cases) - Normalized structures can leverage indexes effectively, improving certain queries.

    Cons of Normalization:

    • More Complex Queries - Requires multiple joins across tables, making queries more complex and potentially slower.
    • Increased Query Execution Time - Joins between tables can lead to performance overhead, especially for large datasets.
    • Difficult to Design and Maintain - Requires careful planning, especially for higher normal forms (3NF, BCNF, etc.).
    • Extra Foreign Keys - Increased number of foreign keys can lead to additional constraints and overhead.
    • Performance Trade-offs in Read-Heavy Applications - Applications that require frequent read operations may suffer due to excessive joins.
    • Denormalization May Be Required for Performance - In some cases, denormalization is needed to improve read performance, requiring extra storage.

    When to Normalize or Denormalize?

    Normalize when:

    • Data consistency and integrity are a priority.
    • The database needs to be scalable and maintainable.
    • The application performs a balanced mix of reads and writes.

    Denormalize when:

    • Performance is a concern, and read operations dominate.
    • The application is analytics-heavy, requiring fewer joins for faster queries.
    • >Data redundancy is acceptable in exchange for speed.

    SQL Object Relational Mapper

    • An SQL Object Relational Mapper (ORM) is a tool that allows developers to interact with a database using an object-oriented paradigm instead of writing raw SQL queries.
    • ORMs provide a high-level abstraction that maps database tables to classes, and rows to objects of those classes.
    • This makes database operations more intuitive for developers who are familiar with object-oriented programming.

    SQL ORMs for different programming languages:

    Python:

    • SQLAlchemy: A powerful and flexible ORM that provides a full suite of enterprise-grade persistence patterns.
    • Django ORM: The built-in ORM of the Django web framework, known for its simplicity and ease of use.

    JavaScript (Node.js):

    • Sequelize: A promise-based ORM for Node.js that supports multiple SQL dialects and provides a robust feature set.
    • TypeORM: An ORM that supports both Active Record and Data Mapper patterns, written in TypeScript.

    Java:

    • Hibernate: A mature and widely used ORM that supports JPA (Java Persistence API) and provides advanced caching and performance features.
    • MyBatis: Unlike traditional ORMs, MyBatis couples objects with stored procedures or SQL statements using XML or annotations.

    Sequelize is the ORM used for this course.

    Benefits of Using an ORM:

    • Productivity: Developers can write less code compared to writing raw SQL, leading to faster development.
    • Maintainability: The code is easier to maintain since it is cleaner and more readable.
    • Portability: ORMs abstract away database-specific details, making it easier to switch between different database systems.
    • Security: ORMs can help prevent SQL injection attacks by using parameterized queries.

    No SQL Datatases

    • Are designed to provide flexible, scalable, and high-performance data storage solutions that differ from the traditional relational (SQL) databases.
    • They are particularly well-suited for handling unstructured or semi-structured data, large volumes of data, and applications requiring high throughput and low latency.

    Advantages of NoSQL Databases:

    • Scalability: Designed to scale out by distributing data across multiple servers.
    • Flexibility: Schema-less design allows for dynamic and evolving data models.
    • Performance: Optimized for high read and write throughput with low latency.
    • Big Data: Capable of handling large volumes of unstructured and semi-structured data.

    MongoDB is the NoSql database used for this course.

    No SQL Object Data Modelling

    • Is similar to Object Relational Mapping (ORM) used in SQL databases, but it's tailored for NoSQL databases.
    • ODMs map objects in your application to documents in a NoSQL database, allowing for more intuitive and structured interaction with the database.

    Key Features of ODMs for NoSQL Databases:

    • Schema Definition: Define the structure of documents and collections using a schema.
    • Data Validation: Ensure data integrity by validating data against the schema before saving.
    • CRUD Operations: Simplified methods for creating, reading, updating, and deleting documents.
    • Query Building: APIs to construct complex queries programmatically.
    • Middleware / Plugins: Support for middleware and plugins to extend functionality.
    • Relationships: Define and manage relationships between documents (e.g., references, embedded documents).

    Benefits of Using ODMs:

    • Ease of Use: Simplifies database interactions by using familiar object-oriented paradigms.
    • Maintainability: Promotes cleaner and more maintainable code.
    • Validation: Ensures data integrity through schema validation.
    • Consistency: Provides a consistent way to interact with the database, making it easier to manage changes.

    Mongoose is the ODM used for this course.

    Queues

    • Are fundamental data structures used extensively in computer science and software engineering to manage a collection of elements in a particular order.
    • They follow the First-In-First-Out (FIFO) principle, where the first element added to the queue is the first one to be removed.
    • Queues are essential for various scenarios, such as task scheduling, managing requests in web servers, handling asynchronous events.

    Applications of Queues:

    • Task Scheduling: Managing tasks in operating systems, job scheduling in servers.
    • Message Queuing: Handling communication between distributed systems (e.g., RabbitMQ, Kafka).
    • Breadth-First Search (BFS): Traversing or searching tree/graph data structures.
    • Caching: Implementing Least Recently Used (LRU) caches.
    • Load Balancing: Distributing tasks across multiple servers.

    Popular Queue Libraries and Services:

    • RabbitMQ: An open-source message broker that implements the Advanced Message Queuing Protocol (AMQP).
    • Apache Kafka: A distributed event streaming platform capable of handling high-throughput and low-latency messaging.
    • Amazon SQS: A fully managed message queuing service that enables decoupling and scaling of microservices, distributed systems, and serverless applications.
    • Redis: An in-memory data structure store that supports list operations, which can be used to implement queues.

    Redis and Rabbitmq are the queues used for this course.
